## 1. System Overview

Pocket SDR is an open-source GNSS software-defined radio (SDR) receiver. It
processes digitized intermediate-frequency (IF) samples from the Pocket SDR FE
hardware frontend and produces:

- GNSS signal acquisition results (code offset, Doppler, C/N0)
- Tracked signal observables (carrier phase, pseudorange, Doppler)
- Decoded navigation data (ephemerides, time, corrections)
- Position/Velocity/Time (PVT) solutions

The software is organized into a layered pipeline:

```
RF Front-end (Pocket SDR FE)
        │  digitized IF samples (int8)
        ▼
IF Data Buffer
        │
        ├─► Acquisition (parallel code search)
        │
        └─► Tracking → Nav Decoding → PVT
```

Supported GNSS systems and signals:

| System   | Signals |
|----------|---------|
| GPS      | L1C/A, L1C-D, L1C-P, L2C-M, L5-I, L5-Q |
| GLONASS  | L1C/A (FDMA), L2C/A (FDMA), L1OCd, L1OCp, L2OCp, L3OCd, L3OCp |
| Galileo  | E1-B, E1-C, E5a-I, E5a-Q, E5b-I, E5b-Q, E6-B, E6-C |
| QZSS     | L1C/A, L1C/B, L1C-D, L1C-P, L1S, L2C-M, L5-I, L5-Q, L5S-I, L5S-Q, L6D, L6E |
| BeiDou   | B1I, B1C-D, B1C-P, B2a-D, B2a-P, B2I, B2b-I, B3I |
| NavIC    | L1-SPS-D, L1-SPS-P, L5-SPS |
| SBAS     | L1C/A, L5-I, L5-Q |

## 2. IF Data Handling

### 2.1 Data Format

The Pocket SDR FE captures IF data in one of the following formats:

| Format | Description |
|--------|-------------|
| I-sampling (real) | Single int8 samples per time step |
| IQ-sampling (complex) | Interleaved int8 pairs: I, Q, I, Q, … |

The `read_data()` function (`sdr_func.py`) reads raw int8 samples from a binary
file and converts them to complex64 (numpy `complex64`) arrays:

- **I-sampling**: `data = int8_sample + 0j`
- **IQ-sampling**: `data = I_sample - j * Q_sample`

The negative sign on Q derives from the down-conversion convention used by the
MAX2771 frontend IC.

### 2.2 Sampling Parameters

Key parameters for IF data processing:

| Parameter | Symbol | Typical Value |
|-----------|--------|---------------|
| Sampling frequency | `fs` | 12 MHz, 24 MHz, 48 MHz |
| IF frequency | `fi` | 0 Hz (zero-IF) or non-zero |
| Sample type | `IQ` | 1 (I-only) or 2 (IQ) |

For zero-IF (`fi = 0`), IQ-sampling is assumed automatically.

### 2.3 GLONASS FDMA Frequency Shift

GLONASS uses FDMA with per-satellite frequency channel numbers (FCN).
The IF frequency is shifted accordingly before correlation:

```
G1CA:  fi_shifted = fi + 0.5625 MHz × FCN
G2CA:  fi_shifted = fi + 0.4375 MHz × FCN
```

## 3. Spreading Code Generation

### 3.1 Code Types

Pocket SDR generates spreading codes for all supported signals in software.
The primary code is a PRN-specific binary sequence (chip values ±1).
Several signal types additionally use a secondary (overlay) code modulated on
top of the primary code to extend coherent integration.

| Signal class | Primary code | Secondary code |
|---|---|---|
| GPS L1C/A | 1023-chip Gold code | None |
| GPS L1C-D | 10230-chip Weil code | 1800-chip overlay |
| GPS L5-I | 10230-chip XB code | 10-chip Neumann-Hofman |
| Galileo E1-B | 4092-chip memory code | None |
| Galileo E5a-I | 10230-chip memory code | 20-chip secondary |
| QZSS L6D/L6E | 10230-chip CSK code | None |
| BeiDou B1C-D | 10230-chip Weil code | 1800-chip overlay |
| GLONASS L1C/A | 511-chip m-sequence | None |

### 3.2 Code Generation Method

Most codes are generated by linear feedback shift registers (LFSRs) or
stored memory tables derived from the respective ICD. The generation
polynomial for each signal follows its official interface control document (ICD).

`sdr_code.py` and `sdr_code_gal.py` implement code generation for all
supported signals.

### 3.3 Code FFT Pre-computation

For the FFT-based correlator (section 4.2), the code DFT (discrete Fourier
transform) is pre-computed once per channel and reused across all Doppler
bins:

```
code_fft = conj(FFT(code_resampled_with_zero_padding))
```

Zero-padding to twice the code length (2N) is optional (`-nz` flag disables
it). Zero-padding enables circular correlation without aliasing.

### 3.4 Code Resampling

When the code chip rate does not divide evenly into the sampling frequency,
the code is resampled to the sample grid. For tracking, a code bank of
`N_CODE = 10` versions with sub-sample offsets is maintained to allow
fractional-chip code offset tracking without re-interpolation at every epoch.

## 4. Signal Acquisition

### 4.1 Parallel Code Search

Signal acquisition searches a two-dimensional space of code offset τ and
Doppler frequency f_d. The search uses a parallel code search algorithm
based on the circular cross-correlation property of the FFT:

**Algorithm:**

For each Doppler bin f_d,i:

1. Mix the IF data buffer with a local carrier at frequency `fi + f_d,i`:
   ```
   data_mixed[n] = data[n] · exp(−j 2π (fi + f_d,i) n / fs)
   ```

2. Compute the circular cross-correlation with the code via FFT:
   ```
   C[τ] = IFFT( FFT(data_mixed) · code_fft )
   ```
   where `code_fft = conj(FFT(code))`.

3. Compute correlation power:
   ```
   P[i, τ] = |C[τ]|²
   ```

The result is a 2D power surface P(f_d, τ).

### 4.2 Non-Coherent Integration

To improve sensitivity, correlation powers from multiple code cycles are
accumulated non-coherently (power summation, not complex summation):

```
P_sum += P[i, τ]   for each code cycle
```

The default non-coherent integration time is `T_ACQ = 10 ms` (approximately
10 code cycles for 1 ms codes). This avoids the navigation data bit
ambiguity while providing ~10 dB integration gain.

### 4.3 Detection and Threshold

After integration, the peak is found:

```
P_max = max(P_sum)
P_ave = mean(P_sum)
C/N0 = 10 · log10((P_max − P_ave) / P_ave / T_code)   [dB-Hz]
```

The signal is declared acquired when `C/N0 ≥ 35 dB-Hz`.

### 4.4 Fine Doppler Estimation

The coarse Doppler bin index is refined by fitting a quadratic polynomial
to the three power values around the peak bin:

```
p(f) = a·f² + b·f + c
f_fine = −b / (2a)
```

This provides sub-bin Doppler accuracy without reducing the bin spacing.

### 4.5 Doppler Search Range

Default Doppler search range: ±5000 Hz.
Search step: `DOP_STEP / T_code = 0.5 / T_code` Hz (0.5 bins per code cycle).

For GLONASS FDMA signals, the search centre is shifted by the FCN frequency
offset before the search.

## 5. Signal Tracking

### 5.1 Correlator Structure

The tracking loop uses a set of correlators at different code phase offsets:

| Correlator | Offset | Purpose |
|------------|--------|---------|
| Prompt (P) | 0 chips | Phase/frequency error discrimination |
| Early (E) | +d/2 chips | DLL discriminator |
| Late (L) | −d/2 chips | DLL discriminator |
| Noise (N) | −80 samples | Noise floor estimate for C/N0 |

Default correlator spacing: `d = 0.25 chips` (`SP_CORR`).

At each epoch (one code cycle), the correlator outputs are computed by
mixing the IF data with the local carrier replica and integrating against
the local code replica:

```
C_k = Σ data[n] · exp(−j 2π fc_k n / fs) · code[n − τ_k]   /   N
```

For QZSS L6D/L6E (CSK modulation), the FFT correlator is used instead (see
section 5.5).

### 5.2 Carrier Tracking: FLL → PLL

The tracking loop uses a two-stage approach:

**Stage 1 – Frequency Lock Loop (FLL)**
Active for the first `T_FPULLIN = 1.0 s` to achieve initial frequency lock.

The FLL discriminator uses the cross/dot product of consecutive P correlator
outputs:

```
dot   = I1·I2 + Q1·Q2
cross = I1·Q2 − Q1·I2
err_freq = atan(cross / dot)    (Costas, for BPSK)
         = atan2(cross, dot)    (non-Costas, for data-less)
```

FLL bandwidth transitions from 5 Hz (wide) to 2 Hz (narrow) at the halfway
point of the pull-in time.

Frequency update:
```
fd += (B_FLL / 0.25) · err_freq / (2π)
```

**Stage 2 – Phase Lock Loop (PLL)**
Activated after FLL pull-in. Uses a second-order PLL.

PLL discriminator:
```
err_phas = atan(QP / IP) / (2π)    (Costas, for BPSK)
         = atan2(QP, IP) / (2π)    (non-Costas)
```

PLL bandwidth: `B_PLL = 5.0 Hz`.
Second-order PLL update (bilinear transform):
```
W = B_PLL / 0.53
fd += 1.4·W·(err_phas − err_phas_prev) + W²·err_phas·T
```

### 5.3 Code Tracking: DLL

A non-coherent Early-minus-Late (EML) delay lock loop (DLL) tracks the code
phase.

Discriminator (normalized):
```
err_code = (|E| − |L|) / (|E| + |L|) / 2 · T_code / code_length   [s]
```

The DLL runs every `N = T_DLL / T_code` epochs (default: 10 epochs for 1 ms
codes) using accumulated E and L magnitudes.

Code phase update:
```
coff -= (B_DLL / 0.25) · err_code · T_code · N
```

DLL bandwidth: `B_DLL = 0.25 Hz`.

**Carrier-aided code tracking:**
The code phase is aided by the tracked carrier Doppler to reduce code
tracking noise:

```
coff -= fd / fc · Δt
```

where `fc` is the carrier frequency and `Δt` is the elapsed time since the
last update.

### 5.4 C/N0 Estimation

C/N0 is estimated using the signal-to-noise ratio of the Prompt correlator
output versus the noise reference correlator (N):

```
C/N0 = 10 · log10(Σ|P|² / Σ|N|² / T_code)   [dB-Hz]
```

The estimate is smoothed with an exponential moving average (α = 0.5) over
`T_CN0 = 1.0 s`.

Signal lock is declared lost when `C/N0 < 32 dB-Hz`.

### 5.5 Secondary Code Synchronization

Pilot signals (e.g., GPS L1C-P, Galileo E1-C, BeiDou B1C-P) modulate the
primary code with a secondary (overlay) code. Pocket SDR detects and removes
this secondary code:

1. The cross-correlation of the P correlator history with the known secondary
   code sequence is computed.
2. Synchronization is declared when the normalized correlation exceeds
   `THRES_SYNC = 0.02`.
3. After synchronization, each P correlator output is multiplied by the
   corresponding secondary code chip.

### 5.6 CSK Demodulation (QZSS L6)

QZSS L6D/L6E uses Code-Shift Keying (CSK) modulation. The transmitted symbol
is encoded as a circular shift of the spreading code. Pocket SDR decodes it
using the FFT correlator:

1. Compute the full circular cross-correlation over the code period via FFT.
2. Interpolate the correlation power profile at ±255 chips around the center.
3. Detect the peak position; the peak shift index gives the CSK symbol value
   (0–255).
4. The correlator output at the peak position is used as the P correlator
   value for subsequent tracking.

## 6. Navigation Data Decoding

### 6.1 Symbol Synchronization

Navigation data bits (symbols) are extracted from the P correlator outputs.
Symbol boundaries are detected by monitoring transitions in the sign of
accumulated P correlator outputs. Once synchronized, each bit is decided by
the sign of the integrated P output over one symbol period.

### 6.2 Frame Synchronization

Each navigation message format has a known preamble or sync word. Pocket SDR
searches for the preamble in the accumulated symbol buffer. On detection,
the frame boundary and data polarity (normal/inverted) are established.

### 6.3 Supported Navigation Messages

| Signal | Message format | Key content |
|--------|---------------|-------------|
| GPS L1C/A | LNAV (25 words × 30 bits) | Ephemeris, almanac, UTC/iono |
| GPS L1C-D | CNAV-2 (subframe 2 & 3) | Ephemeris |
| GPS L2C-M | CNAV | Ephemeris, clock corrections |
| GPS L5-I | CNAV | Ephemeris, clock corrections |
| Galileo E1-B | I/NAV (page pairs) | Ephemeris, iono, UTC |
| Galileo E5b-I | I/NAV | Ephemeris |
| Galileo E6-B | HAS (High Accuracy Service) | PPP corrections |
| QZSS L1C/A | LNAV | Ephemeris (QZSS-specific) |
| QZSS L6D | CLAS message (Reed-Solomon) | Centimetre-level augmentation |
| BeiDou B1I | D1/D2 NAV | Ephemeris, almanac |
| BeiDou B1C-D | CNAV-1 | Ephemeris |
| BeiDou B2a-D | CNAV-2 | Ephemeris |
| GLONASS L1C/A | GLONASS NAV (strings) | Ephemeris, almanac |
| NavIC L5-SPS | IRNSS NAV | Ephemeris |

## 7. Error Correction

### 7.1 Convolutional Coding / Viterbi Decoding

Many GNSS navigation messages (GPS L5, Galileo E1-B/E5, BeiDou B1C) use
rate-1/2, constraint-length-7 convolutional channel coding. Pocket SDR
decodes these using the Viterbi algorithm implemented in the `libfec` external
library.

**Code parameters:**

| Parameter | Value |
|-----------|-------|
| Rate | 1/2 |
| Constraint length | K = 7 |
| Generator polynomials | G1 = 0x4F, G2 = 0x6D |

### 7.2 CRC Checking

Navigation frames include CRC (cyclic redundancy check) fields that are
verified after Viterbi decoding. Only frames passing CRC are accepted.

### 7.3 LDPC Decoding (BeiDou B1C, B2a, B2b)

BeiDou new-generation signals use Low-Density Parity-Check (LDPC) codes.
Pocket SDR provides two decoders:

- **Full LDPC decoder** (`sdr_ldpc.py`): Uses the `LDPC-codes` external
  library (sum-product / belief propagation algorithm).
- **NB-LDPC decoder** (`sdr_nb_ldpc.py`): A compact non-binary LDPC decoder
  optimized for the specific code parameters of B1C, B2a, and B2b subframes.

### 7.4 BCH Error Correction (BeiDou B1I)

BeiDou B1I uses BCH(15, 11, 1) error correction for certain fields. Pocket SDR
implements this via a pre-computed error correction lookup table
(`BCH_CORR_TBL`, `sdr_nav.py`) covering all single-bit error patterns.

### 7.5 Reed-Solomon Decoding (QZSS L6)

QZSS L6D CLAS data uses Reed-Solomon error correction applied across message
frames. Pocket SDR decodes this using the `libfec` library's RS(255, 223)
implementation.

## 8. PVT Generation

### 8.1 Overview

The PVT module (`sdr_pvt.c`) generates position, velocity, and time solutions
using RTKLIB as the positioning engine. PVT is computed at 1 Hz epochs.

### 8.2 Observation Data

At each epoch, pseudorange and carrier phase observables are assembled from
all locked channels:

- **Pseudorange**: derived from the tracked code phase offset and receiver time.
- **Carrier phase**: integrated from the accumulated Doppler and the
  current carrier phase.
- **Doppler**: from the tracked carrier frequency offset (fd).
- **C/N0**: from the C/N0 estimator (section 5.4).

An elevation mask of 15° is applied to reject low-elevation satellites.

### 8.3 Positioning Engine

PVT uses RTKLIB's least-squares single-point positioning algorithm. The
solution accounts for:

- Satellite positions and velocities from decoded ephemerides
- Ionospheric delay (Klobuchar model for GPS/QZSS, Galileo NeQuick broadcast)
- Tropospheric delay (Saastamoinen model)
- Satellite clock corrections from navigation data
- Relativistic corrections

### 8.4 Output Formats

PVT results are output in the following formats:

| Format | Description |
|--------|-------------|
| NMEA 0183 | GGA, RMC, GSA, GSV sentences (standard marine/mapping format) |
| RTCM3 | RTCM 10403.x binary format for differential corrections |
| Internal log | CSV-format `$POS`, `$OBS`, `$SAT` log records |

## 9. Snapshot Positioning

### 9.1 Principle

Snapshot positioning (`pocket_snap.py`, `pocket_snap`) computes a position
from a single short capture of IF data (typically a few milliseconds), without
requiring continuous tracking or navigation data download.

It uses a coarse time and approximate position to:

1. Predict the expected code offset for each visible satellite.
2. Acquire signals to measure the actual code offsets.
3. Compute position from the pseudorange differences (code offset residuals).

### 9.2 Algorithm

1. **Signal acquisition**: Run parallel code search over the snapshot buffer
   for each candidate satellite and signal.
2. **Pseudorange prediction**: Using the known approximate receiver position
   and time, compute predicted pseudoranges from broadcast ephemerides.
3. **Residual computation**: Compute pseudorange residuals between measured
   and predicted values.
4. **Position solution**: Solve for position using RTKLIB's positioning
   engine with the residual pseudoranges.

This approach requires an initial position estimate (within ~150 km) and
approximate time (within ~1 minute).

## 10. Receiver Channel State Machine

Each receiver channel (`sdr_ch.py`) is an independent state machine with
three states:

```
         ┌──────────────────────────────────────────┐
         │                                          │
    ┌────▼────┐  C/N0 ≥ threshold  ┌────────────┐  │ C/N0 < threshold
    │  IDLE   │───────────────────►│   SRCH     │  │
    └─────────┘                    └────────────┘  │
         ▲                              │           │
         │                    signal    │           │
         │                    found     ▼           │
         │                        ┌────────────┐   │
         └────────────────────────│   LOCK     │───┘
              signal lost         └────────────┘
```

| State | Description |
|-------|-------------|
| `IDLE` | Waiting for the next acquisition trigger |
| `SRCH` | Running parallel code search (acquisition) |
| `LOCK` | Running tracking loops (DLL/FLL/PLL) and decoding navigation data |

### State Transitions

- **IDLE → SRCH**: Triggered externally by the receiver scheduler.
- **SRCH → LOCK**: Signal found (C/N0 ≥ 35 dB-Hz after `T_ACQ` integration).
- **SRCH → IDLE**: Signal not found after `T_ACQ` integration.
- **LOCK → IDLE**: C/N0 drops below 32 dB-Hz (signal lost).

### Timing Parameters

| Parameter | Symbol | Default | Description |
|-----------|--------|---------|-------------|
| Acquisition integration | `T_ACQ` | 10 ms | Non-coherent integration for acquisition |
| FLL pull-in | `T_FPULLIN` | 1.0 s | Duration of FLL before switching to PLL |
| Nav pull-in | `T_NPULLIN` | 1.5 s | Delay before starting nav data decoding |
| DLL integration | `T_DLL` | 10 ms | Non-coherent integration for DLL |
| C/N0 averaging | `T_CN0` | 1.0 s | Averaging window for C/N0 estimator |
| Lock threshold | `THRES_CN0[0]` | 35 dB-Hz | C/N0 to declare acquisition success |
| Lost threshold | `THRES_CN0[1]` | 32 dB-Hz | C/N0 below which tracking is lost |
